================================================================
LLMPlayer buy-decision prompt — full transcript
================================================================

This file shows the exact prompt fed to the LLM in `agents.LLMPlayer`,
laid out in the same conversation format the chat template produces.
Default model: Qwen/Qwen2.5-1.5B-Instruct.
Generation: greedy (do_sample=False), max_new_tokens=160, with
<|im_end|> and <|endoftext|> as additional EOS tokens.

The conversation is six messages:
  - 1 system message
  - 4 few-shot user/assistant exchanges
  - 1 runtime user message describing the actual decision

The model produces an assistant response that:
  1. Begins with an ECHO block copying all 10 STATE fields verbatim.
  2. Then a REASON sentence.
  3. Then "ANSWER: BUY" or "ANSWER: PASS".

The parser scans for the FIRST 'ANSWER: BUY' or 'ANSWER: PASS' (the
ECHO block contains no 'ANSWER:' line, so the answer line is
unambiguously the model's decision).

A separate echo validator (LLMPlayer._check_echo) compares each echoed
value to the ground-truth STATE; on any mismatch it triggers up to
MAX_RETRIES=4 retries with corrective follow-up turns. The final
attempt's answer is the decision used by the simulator.

Build, trade, and jail decisions are NOT routed through the LLM —
those keep the base Player logic to keep latency manageable.

----------------------------------------------------------------
[ system ]
----------------------------------------------------------------
You are a strategic Monopoly player who decides whether to buy
properties. Your goal is to win by bankrupting opponents through
rent. You must NOT mindlessly buy every property: passing is correct
when (a) cash is dangerously low, (b) the property is in a group an
opponent already dominates and you have no path to monopoly, or (c)
the rent return is poor relative to the cost. Equally, buying is
correct when the property advances a group you already own most of,
or when it denies the group to an opponent who would otherwise
complete it.

You receive a STATE block with 10 fields. Before reasoning, you
MUST emit an ECHO block that copies all 10 STATE fields verbatim
(numeric values must equal STATE; string values must equal STATE).
The ECHO block is checked deterministically — if it differs from
STATE in any field you will be asked to redo the answer. Only after
the ECHO block do you write REASON and ANSWER. Do not invent any
number or name not present in STATE. A purchase ONLY 'completes a
monopoly' when you_own_in_group + 1 == group_size; do not claim
completion otherwise.

Reply in EXACTLY this format and order:
  ECHO:
    cash: $<value>
    property: <name>
    group: <name>
    cost: $<value>
    base_rent: $<value>
    you_own_total: <value>
    you_own_in_group: <value>
    group_size: <value>
    opp_own_in_group: <value>
    opp_own_total: <value>
  REASON: <one short sentence citing only STATE numbers>
  ANSWER: BUY
or
  ... (same ECHO block) ...
  REASON: <one short sentence>
  ANSWER: PASS
Do not output anything else.

================================================================
Pre-filter before the LLM is even called
================================================================

LLMPlayer._should_buy applies three cheap rule-based checks before
constructing the prompt. Anything that fails one of these returns
PASS without an LLM call:

  - if cost > current_cash:                      PASS  (can't afford)
  - if cash - cost < settings.unspendable_cash:  PASS  (cash floor)
  - if prop.group in settings.ignore_groups:     PASS  (player policy)

This typically eliminates 60-80% of would-be LLM calls per game.


================================================================
Decision parsing
================================================================

The model's response is parsed by LLMPlayer._parse_decision /
_parse_decision_traced:

  1. Find the FIRST 'ANSWER: BUY' or 'ANSWER: PASS' (case-insensitive)
     in the response. The earlier of the two wins.
  2. If neither structured tag appears, fall back to the LAST
     occurrence of 'BUY' or 'PASS' anywhere in the response.
  3. If neither token appears, default to BUY.

The first-occurrence rule is critical because small models often
hallucinate a fake new user turn after their answer. The first-
occurrence rule guarantees we read the model's actual decision.


================================================================
ECHO validator (added 2026-04-28T18; supersedes the regex detector)
================================================================

LLMPlayer._check_echo parses the assistant response for "key: value"
lines and compares each of the 10 expected fields against ground-
truth STATE:

  Numeric fields  : cash, cost, base_rent,
                    you_own_total, you_own_in_group, group_size,
                    opp_own_in_group, opp_own_total
                    (parsed with int(), tolerating leading $ and -)
  String fields   : property, group
                    (case-sensitive equality)

Any missing field, unparseable value, or mismatch produces an issue
string like:

  "echo mismatch on 'cash': model echoed 80, STATE.cash=1500"

These strings drop directly into the corrective user turn for the
retry. The retry loop runs up to MAX_RETRIES=4 times (5 total
attempts per decision); each retry's tail appends EVERY prior
assistant response plus a corrective user turn after the latest, so
the model sees its accumulated history of wrong answers.

The FINAL attempt's parsed answer is the decision used by the
simulator. The full attempt list is logged under "echo_attempts" in
the per-game JSONL; original (attempt 0) and final-attempt fields
are also surfaced under raw_response/reason_text/parsed and
retry_response/retry_reason_text/retry_parse_path for back-compat
with the 2026-04-28T16 schema.

The earlier regex+predicate detector (LLMPlayer._check_reason_consistency)
remains in agents.py but is no longer called from the logging path.
The echo validator strictly dominates it.
